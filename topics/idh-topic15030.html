After going through the file, it seems that probably, this test escapee issue may have occured between 08/July and 09/July.

For test escapee issue, basically there are 2 scenerios. Device drop into socket before testing and device drop/stuck in socket after testing.

I believe the latter scenerio may have been the cause of the test escapee.

Before Test

-Device drop from either Index Arm 1 or Index Arm 2

-JAM0303 "Device drop error (Arm 1)" or JAM0304 "Device drop error (Arm 2)" - Only SKIP available

-MES0305 "Check again! Device drop error (Arm 1)" or MES0306 "Check again! Device drop error (Arm 2)" - Only SKIP is available

-Device drop during this time, almost 99% the dropped device will not be dropped into socket exactly testing position, sitted nicely like hand test...

-Yes, device may be dropped into socket but it may lead to device cracked due to subsequent testing and open contact due to cracked device or maybe torque error.

After Test

-Device drop error from Index Arm 1 or Index Arm 2 (or device stuck in socket after testing)

-JAM0303 "Device drop error (Arm 1)" or JAM0304 "Device drop error (Arm 2)" - Only SKIP available

-Affected index arm will proceed to check device exist in socket by vaccum sensor:

-If device stuck in socket (test escapee), JAM0315 "Device stack error (double device)" - RETRY and SKIP available

-If RETRY, index arm will always go down into socket to check by vacuum untill device is removed

-IF SKIP, index arm will not check and proceed normal operation. This is where test escapee happens if SKIP is pressed without removing device.

SKIP - Purpose is to skip error. In case it is due to vaccum sensor ON because contactor touching socket/pogo pin and not device stuck in socket. This happens when threshold is set too low for example.

RETRY - Index Arm will always be checking socket untill device is removed or vaccum sensor OFF.

With SKIP removed, only RETRY is available so operator can only RETRY.

SKIP removed does not mean remove SKIP button. It is only to remove the SKIP option for JAM0315.

So initial drop error after testing, first SKIP+START for drop error, index arm will check socket.

If there is device in socket, first JAM0315 will trigger only RETRY so in this case, operator has to RETRY untill device is removed.

Which means on JAM0315, operator can only RETRY+START untill error is cleared..no SKIP available..

Is this ok?

If you prefer to have SKIP option remain, but after SKIP+START and check socket, there is no meaning as this is same as RETRY+START.

Because purpose of SKIP is to skip error when there is no device in socket but still JAM0315 as mentioned in the first line of my email.

Which is why I recommend to consider just don't allow SKIP in the case of JAM0315.
